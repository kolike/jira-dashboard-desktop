# Технический аудит Jira Fast Watcher

Дата аудита: 2026-04-22

## Краткий вывод
Приложение уже решает основную бизнес-задачу (мониторинг очередей Jira + уведомления + мини-дашборд), но находится в стадии **быстрого монолита**: почти вся логика сосредоточена в `main.py`, из-за чего растут риски регрессий, тяжело тестировать и дорого добавлять функции.

Ниже — приоритетный план доведения до "идеала".

---

## Приоритет P0 (исправить в первую очередь)

1. **Разделить код по модулям**.
   - Сейчас UI, сетевой слой, состояние, уведомления и бизнес-логика в одном файле.
   - Рекомендация: минимум вынести в:
     - `jira_client.py` (API Jira),
     - `state_store.py` (config/state IO),
     - `notifier.py` (toast/tray),
     - `windows/` (Qt окна),
     - `app_controller.py` (оркестрация).

2. **Убрать платформенную зависимость в import-time**.
   - Вызов `ctypes.windll...` выполняется при импорте модуля и потенциально ломает запуск вне Windows.
   - Рекомендация: вызывать это только при `sys.platform == "win32"` и оборачивать в безопасный helper.

3. **Секреты и токены**.
   - PAT хранится в `config.json` в открытом виде.
   - Рекомендация: хранение токена в Windows Credential Manager (`keyring`) + в конфиге только ключ/alias.

4. **Thread-safety и жизненный цикл потоков**.
   - Сейчас создаются ad-hoc `threading.Thread`, есть ручной флаг `_check_in_progress`.
   - Рекомендация: перейти на `QThreadPool + QRunnable` или единый worker-queue с таймаутами и отменой.

## Приоритет P1 (качество и поддерживаемость)

1. **Валидация настроек до сохранения**.
   - Проверять URL, минимальную длину токена, синтаксис JQL (через lightweight тестовый запрос).

2. **Централизованная обработка ошибок Jira API**.
   - Добавить единый декоратор/функцию нормализации ошибок (`401`, `403`, `429`, `5xx`) и дружелюбные сообщения для UI.

3. **Логирование в файл + ротация**.
   - Сейчас логи в памяти и stdout.
   - Рекомендация: `RotatingFileHandler` (например 5x2MB) + экспорт логов из GUI.

4. **Typed-модели для Jira issue**.
   - Вместо большого количества `dict[str, Any]` — `dataclass`/`pydantic`-модели для ключевых полей.

5. **Устранить дубли импорта/неиспользуемые части**.
   - В файле есть повторные импорты Qt-классов и исторические методы, которые повышают шум.

## Приоритет P2 (UX и продукт)

1. **Мягкая деградация интерфейса**.
   - Добавить индикатор "нет связи / ретрай через N сек", а не только всплывающее сообщение.

2. **Фильтрация/поиск в списках задач**.
   - По ключу, региону, статусу, автору.

3. **Более информативные карточки задач**.
   - SLA/возраст заявки, приоритет, время с момента создания.

4. **Управление уведомлениями**.
   - Отдельные тумблеры: RED/BLUE, quiet hours, лимит повторов.

---

## Тестовая стратегия (чтобы реально удерживать качество)

1. **Unit-тесты**:
   - парсинг региона,
   - сортировка задач,
   - merge config/state,
   - форматирование текста карточек.

2. **Интеграционные тесты API**:
   - через `responses`/`pytest-httpx` мокать Jira API.

3. **Smoke e2e для GUI**:
   - базовый запуск, сохранение настроек, ручной check.

4. **Quality gates в CI**:
   - `ruff`, `black`, `mypy` (или pyright), `pytest`.

---

## Целевая архитектура (к чему идти)

- `core/`: доменные модели + use-cases.
- `infra/`: Jira HTTP client, storage, notifier.
- `ui/`: Qt окна и виджеты.
- `app/`: wiring/DI, lifecycle.

Это даст:
- проще onboarding,
- ниже цена изменений,
- легче покрытие тестами,
- меньше случайных багов при добавлении фич.
